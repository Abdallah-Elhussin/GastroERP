## التفاصيل

### تسمية الملفات

| النوع | النمط | مثال |
|-------|-------|------|
| فهرس | `00 - Documentation Index.md` | المرجع الرئيسي |
| أساسيات | `01 - Project Overview.md` | نظرة عامة |
| مراحل | `Phases/29 - Reporting & Analytics.md` أو `29 — … .md` | توثيق Phase |
| وحدات | `Modules/Reporting.md` | ملخص وحدة |
| ADR | `Architecture/ADR-NNN-Title.md` | قرارات معمارية |

**قواعد:**

- استخدم ` - ` (مسافة-شرطة-مسافة) أو ` — ` (em dash) بشكل ثابت داخل المجلد.
- تجنّب المسافات الزائدة وأسماء غير وصفية.
- لا تكرر نفس المحتوى في ملفين — استخدم رابطاً لملف واحد canonical.

### هيكل المستند

كل ملف Phase أو Module يبدأ بـ:

```markdown
# عنوان الوثيقة

## الهدف

...

## المحتويات

(اختياري للملفات الطويلة)

## التفاصيل

...

## Build Status

0 Errors · 0 Warnings

## الملاحظات

...

## النتيجة
```

### مستويات العناوين

| المستوى | الاستخدام |
|---------|-----------|
| `#` | عنوان الملف (واحد فقط) |
| `##` | أقسام رئيسية |
| `###` | أقسام فرعية |
| `####` | تفاصيل نادرة |

### الجداول والقوائم

- استخدم جداول Markdown للحالة، APIs، Migrations.
- استخدم قوائم مرقّمة للخطوات، ونقطية للميزات.
- حافظ على محاذاة الأعمدة في المصدر قدر الإمكان.

### كتل الكود

- أوامر Shell: `powershell` أو `bash`
- مسارات C#: مسار الملف عند الاقتباس
- لا تُخفِ أجزاء الأوامر الحرجة في الوثائق التشغيلية

### الروابط

- روابط نسبية داخل `Document/`
- ترميز URL للمسافات: `%20` أو `%20—%20`
- تحقق من الروابط عند إعادة التسمية

### حالة Build

في جميع ملفات Phases المكتملة:

```markdown
## Build Status

0 Errors · 0 Warnings
```

### تسمية الوحدات

| الاسم الرسمي | لا تستخدم |
|--------------|-----------|
| Reporting & Analytics | Reporting only / BI only |
| Workflow Engine | Approval only |
| Finance & Accounting | Accounting only |
| Human Resources | HR only (في العناوين العربية: الموارد البشرية) |
| Inventory Management | Stock only |
